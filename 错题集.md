## 模块 1：JavaScript 核心基础

### 题目：JavaScript 隐式类型转换

以下表达式的结果分别是什么？请说明类型转换的过程。

```javascript
console.log([] == ![]);           // ?
console.log({} + []);             // ?
console.log(true + false);        // ?
console.log('1' - 1);             // ?
console.log([] + {});             // ?
console.log(NaN === NaN);         // ?
```

#### 答案与解析

```javascript
console.log([] == ![]);           // true
console.log({} + []);             // "[object Object]"
console.log(true + false);        // 1
console.log('1' - 1);             // 0
console.log([] + {});             // "[object Object]"
console.log(NaN === NaN);         // false
```

#### 核心转换规则

1. **`[] == ![]`** → `true`
   - `![]`：`[]` 是 truthy，所以 `![]` 得 `false`
   - `[] == false`：布尔转数字 → `[] == 0`
   - `[]` 转数字：先 `toString()` 得 `""`，再 `Number("")` 得 `0`
   - `0 == 0` → `true`

2. **`{} + []`**（表达式上下文中）→ `"[object Object]"`
   - 对象 `toString()` 得 `"[object Object]"`
   - 数组 `toString()` 得 `""`
   - 字符串拼接得 `"[object Object]"`

3. **`true + false`** → `1`
   - 双方都转数字：`1 + 0 = 1`

4. **`'1' - 1`** → `0`
   - `-` 运算符将两边都转数字：`1 - 1 = 0`

5. **`[] + {}`** → `"[object Object]"`
   - 同第 2 题，两边都调 `toString()` 后拼接

6. **`NaN === NaN`** → `false`
   - IEEE 754 规定 NaN 不等于自身
   - 判断 NaN 用 `Number.isNaN()` 或 `Object.is()`

#### 记忆口诀

- **`==` 比较**：`null/undefined` 只互等不转；布尔优先转数字；对象先转原始值
- **`+` 运算**：有一方是字符串就拼接，否则转数字
- **`- * /` 运算**：一律转数字

#### 对象转原始值规则（ToPrimitive）

1. 先调用 `Symbol.toPrimitive` 方法（如果存在）
2. 再调用 `valueOf()`，如果返回原始值则使用
3. 再调用 `toString()`，如果返回原始值则使用
4. 都无法得到原始值则抛出 TypeError

### 题目：手写 Promise.all

#### 问题

手写实现 `Promise.all`，要求：
- 接收一个可迭代对象，所有 Promise 成功时返回结果数组
- 任一 Promise 失败时立即返回该失败原因
- 注意处理非 Promise 值的情况

#### 易错点

1. 必须返回 Promise，不能返回普通数组
2. 结果必须按输入顺序排列，用索引赋值而非 `push`
3. 失败判断不能在同步循环中检查异步结果
4. 需要计数器判断所有 Promise 是否全部完成

#### 正确实现

```javascript
Promise.myAll = function (promises) {
  return new Promise((resolve, reject) => {
    const results = [];
    let count = 0;
    const len = promises.length;

    if (len === 0) return resolve(results);

    promises.forEach((p, i) => {
      Promise.resolve(p).then(res => {
        results[i] = res;
        if (++count === len) resolve(results);
      }).catch(reject);
    });
  });
};
```

### 题目：hasOwnProperty 与原型链

#### 问题

```javascript
function Person(name) { this.name = name; }
Person.prototype.sayHi = function () { return 'Hi, ' + this.name; };

const p = new Person('Tom');
console.log(p.hasOwnProperty('name'));    // true
console.log(p.hasOwnProperty('sayHi'));   // false
```

#### 易错点

- `hasOwnProperty` 只检查**实例自身（own）的属性**，不沿原型链查找
- `name` 是构造函数里通过 `this.name` 直接挂在实例上的，是 own property
- `sayHi` 定义在 `Person.prototype` 上，实例通过原型链继承访问到，不是自身属性

## 模块 2：Vue 框架原理

### 题目：虚拟 DOM Diff 算法

#### 问题

1. Vue 2 和 Vue 3 的 Diff 算法有什么区别？
2. Vue 3 引入了哪些优化？
3. key 的作用是什么？为什么不推荐用 index 作为 key？

#### 答案与解析

##### Vue 2 的 Diff（双端比较）

- 四个指针：旧头、旧尾、新头、新尾
- 每轮比较：旧头 vs 新头、旧尾 vs 新尾、旧头 vs 新尾、旧尾 vs 新头
- 都没匹配到就遍历旧节点查找
- 时间复杂度 O(n)

##### Vue 3 的 Diff 优化

**a) 最长递增子序列（LIS）**
- 先从头尾去掉相同节点
- 对剩余节点通过 LIS 算法找出不需要移动的子序列
- 只移动不在 LIS 中的节点，减少 DOM 操作

**b) 静态提升**
- 静态节点只创建一次，后续复用，不参与 Diff

**c) PatchFlags（补丁标记）**
- 编译时标记动态节点的类型（文本、class、style 等）
- Diff 时只检查带标记的动态部分

**d) Block Tree**
- 每个模板按结构分块（Block），只收集动态子节点
- 更新时遍历扁平的动态节点数组，不遍历整棵树

##### key 的作用

key 是 Diff 时判断新旧节点是否"同一个"的唯一标识。

**用 index 做 key 的问题：** 列表 `[A, B, C]` 在头部插入 D 变 `[D, A, B, C]`：
- key=0: A→D，key=1: B→A，key=2: C→B，新增 C → 所有节点被错误更新
- 用唯一 id 做 key：A→A、B→B、C→C，只需一次 DOM 插入（D）

### 题目：Vue 2 响应式原理（深度不足）

#### 需要掌握的完整流程

**Object.defineProperty 拦截机制：**
- 初始化时递归遍历 data 所有属性，逐个用 defineProperty 转为 getter/setter
- 这就是为什么大型对象初始化较慢

**依赖收集完整流程：**
```
渲染 Watcher 创建 → 设置 Dep.target = 当前 Watcher
    → 执行 render 函数 → 触发属性 getter
    → getter 中执行 dep.depend()
    → Watcher 被加入 Dep 的 subs 数组

属性变更 → 触发 setter → dep.notify()
    → 遍历 subs 中所有 Watcher → 执行 update()
    → Watcher 重新求值 → 触发重新渲染
```

核心桥梁：`Dep.target` 全局变量

**defineProperty 的局限性：**
- 无法检测属性新增/删除（需 Vue.set / Vue.delete）
- 必须逐个遍历属性，初始化性能差
- 数组下标修改无法拦截，只能 hack 七种数组方法

**Proxy 的优势：**
- 对象级别拦截，无需逐个遍历属性
- 原生支持属性新增/删除检测
- 支持 13 种拦截操作（has、deleteProperty、ownKeys 等）

### 题目：Vue 组件生命周期

#### Vue 2 → Vue 3 生命周期变化

| Vue 2（选项式） | Vue 3（选项式） | Vue 3（组合式） |
|---|---|---|
| beforeCreate | beforeCreate | setup() 替代 |
| created | created | setup() 替代 |
| beforeMount | beforeMount | onBeforeMount |
| mounted | mounted | onMounted |
| beforeUpdate | beforeUpdate | onBeforeUpdate |
| updated | updated | onUpdated |
| beforeDestroy | **beforeUnmount** | onBeforeUnmount |
| destroyed | **unmounted** | onUnmounted |

关键变化：
- `beforeCreate` / `created` 被 `setup()` 取代
- `destroyed` 改名为 `unmounted`
- 组合式 API 全部加 `on` 前缀

#### 父子组件挂载和销毁顺序

**挂载（从内到外）：**
```
父 beforeMount → 子 beforeMount → 子 mounted → 父 mounted
```

**销毁（从内到外）：**
```
父 beforeUnmount → 子 beforeUnmount → 子 unmounted → 父 unmounted
```

#### onMounted vs nextTick

- **`onMounted`**：组件 DOM 挂载完成后触发，只触发一次
- **`nextTick`**：下次 DOM 更新循环结束后执行回调，每次数据变更后都可用
- 核心区别：`onMounted` 是生命周期钩子，`nextTick` 是异步 DOM 更新工具

### 题目：computed 和 watch 底层实现

#### computed 实现原理

computed 本质是一个**带缓存的 effect**：

```javascript
function computed(getter) {
  let value
  let dirty = true  // 脏标记

  const runner = effect(getter, {
    lazy: true,              // 不立即执行
    scheduler: () => {       // 依赖变化时不重新执行，只标记为脏
      dirty = true
      trigger(obj, 'value')  // 通知依赖 computed 的其他 effect
    }
  })

  const obj = {
    get value() {
      if (dirty) {
        value = runner()  // 脏了才重新计算
        dirty = false
      }
      track(obj, 'value')  // 收集依赖此 computed 的 effect
      return value
    }
  }
  return obj
}
```

关键：`lazy: true` 创建时不执行；`scheduler` 依赖变化只设脏标记（惰性求值）；读取 `.value` 时才判断是否需要重新计算。

#### watch 实现原理

watch 本质是一个**带 scheduler 的 effect**：

```javascript
function watch(source, cb) {
  const getter = () => traverse(source)  // 深度遍历所有属性，收集依赖
  let oldValue

  const runner = effect(getter, {
    scheduler: () => {          // 依赖变化时触发 scheduler
      const newValue = runner()
      cb(newValue, oldValue)    // 执行回调
      oldValue = newValue
    }
  })

  oldValue = runner()  // 首次执行，收集依赖
}
```

#### computed vs watch 本质区别

| | computed | watch |
|---|---|---|
| effect 行为 | 依赖变化 → 标脏 → 惰性求值 | 依赖变化 → 立即触发 scheduler → 执行回调 |
| 返回值 | 返回计算结果 | 不关心返回值，执行副作用 |
| 缓存 | 有（dirty 标记） | 无 |

### 题目：Vue Router 核心原理（深度不足）

#### Hash 模式
- URL 中 `#` 后的内容变化不会触发页面请求
- 通过 `hashchange` 事件监听路由变化

#### History 模式
- 使用 `history.pushState()` / `history.replaceState()` 修改 URL 且不刷新页面
- `pushState` 不会触发 `popstate` 事件，Vue Router 需要自己拦截这两个方法
- `popstate` 只在浏览器前进/后退时触发

#### 路由切换不刷新页面的完整流程
```
URL 变化 → 路由监听（hashchange / 拦截 pushState + popstate）
    → Router 根据路由表匹配当前路径
    → 找到对应的路由记录（matched route）
    → 更新 <router-view> 渲染对应组件
```

#### History 模式生产环境部署
nginx 配置所有路由 fallback 到 `index.html`：
```nginx
location / {
  try_files $uri $uri/ /index.html;
}
```
原因：History 模式 URL 是真实路径，直接访问或刷新时服务器会查找对应文件，找不到则 404。需要 fallback 到 `index.html` 交给前端路由处理。

### 题目：Pinia vs Vuex（了解不足）

#### Pinia 相比 Vuex 的优势

| 特性 | Vuex | Pinia |
|---|---|---|
| 修改状态 | 必须 mutation（同步）+ action（异步） | 直接修改或通过 action，没有 mutation |
| 模块化 | 需要 modules + namespaced，嵌套结构 | 每个 store 独立，天然模块化 |
| TypeScript | 类型推断困难 | 原生 TS 支持，自动类型推断 |
| 体积 | ~6KB | ~1.5KB |

#### Pinia 响应式原理

Pinia store 本质是 reactive 对象：
- **state** → 底层是 `reactive()` 创建的响应式对象
- **getters** → 底层是 `computed()`
- **actions** → 普通函数，`this` 绑定到 store 实例

完全复用 Vue 的响应式系统，没有自己实现一套。

#### 组件外使用 Pinia

可以在任意 JS/TS 文件中使用，常用于 axios 拦截器、工具函数等：
```javascript
import { useUserStore } from '@/stores/user'
const userStore = useUserStore()
```
注意：需确保 Pinia 已通过 `app.use(createPinia())` 安装。

## 模块 3：TypeScript 与工程化

### 题目：type vs interface（部分不准确）

#### type 能做但 interface 不能的
- 基本类型别名：`type ID = string | number`
- 联合类型：`type Status = 'active' | 'inactive'`
- 元组：`type Pair = [string, number]`
- 映射类型：`type Readonly<T> = { readonly [K in keyof T]: T[K] }`
- 条件类型：`type IsString<T> = T extends string ? true : false`

#### interface 能做但 type 不能的
- 声明合并（同名的 interface 自动合并）

#### 易错点
- **两者都支持泛型**，不存在"泛型只能用 interface"的说法
- 区分原则：对象形状/可扩展 API → `interface`；联合/映射/条件/元组 → `type`
- 团队统一风格即可

#### 声明合并的实际用途
扩展第三方库类型：
```typescript
declare module 'vue-router' {
  interface RouteMeta {
    requiresAuth?: boolean
    title?: string
  }
}
```

### 题目：Vite 核心原理（不了解）

#### 开发模式：基于浏览器原生 ESM
- 不预先打包，浏览器按需请求模块，Vite 即时编译返回
- 冷启动极快：无论项目多大，只编译当前请求的模块
- HMR 基于模块粒度，修改一个文件只需重新编译该文件

#### 生产模式：基于 Rollup 打包
- 使用 Rollup 进行完整打包（Tree Shaking、代码分割、压缩）
- 不用原生 ESM 的原因：请求瀑布流性能差、无法优化、兼容性问题

#### 依赖预构建（Dependency Pre-bundling）
启动时用 esbuild 预处理 `node_modules`：
- CJS/UMD → ESM 转换
- 减少请求数（如 lodash-es 有 600+ 子模块）
- esbuild 用 Go 编写，比 JS 工具快 10-100 倍
- 结果缓存到 `node_modules/.vite/`

#### Vite vs Webpack

| | Webpack | Vite |
|---|---|---|
| 开发模式 | 预先打包所有模块再服务 | 不打包，按需请求 |
| 冷启动 | 项目越大越慢 | 几乎不受项目大小影响 |
| HMR | 全量重新构建 | 只重新编译修改的模块 |
| 生产构建 | 自身打包 | 使用 Rollup |
| 预处理 | JS 工具链（babel/loader） | esbuild（Go，极快） |

核心区别：Webpack 是"先打包再服务"，Vite 是"先服务，按需编译"。

### 题目：Tree Shaking（深度不足）

#### 完整流程
```
静态分析 ESM 的 import/export
    → 标记哪些 export 被实际使用
    → 未被使用的 export 标记为 unused
```

3. 模块有副作用：`import './polyfill'`
4. package.json 未标记 `sideEffects: false`
5. IIFE（立即执行函数）

## 模块 4：浏览器与网络

### 题目：从输入 URL 到页面渲染的全过程（不了解）

#### 网络请求阶段

1. **DNS 解析**：域名 → IP（浏览器缓存 → OS 缓存 → 本地 DNS → 根域名 → 顶级域名 → 权威域名）
2. **TCP 三次握手**：SYN → SYN+ACK → ACK
3. **TLS 握手**（HTTPS 时）：协商加密密钥、验证证书
4. **HTTP 请求响应**：发送请求 → 服务器返回 HTML

#### 渲染阶段

1. **解析 HTML → DOM 树**：字节流 → Token → DOM 节点 → DOM 树
2. **解析 CSS → CSSOM 树**：CSS 规则 → CSSOM 树
3. **合并 → 渲染树**：DOM + CSSOM → 渲染树（不含 head、display:none 元素）
4. **布局（Layout/Reflow）**：计算每个节点的位置和大小
5. **绘制（Paint）**：将渲染树转为屏幕像素
6. **合成（Composite）**：多个图层由 GPU 合成最终画面

#### JS 对渲染的影响

- 普通 `<script>`：阻塞 DOM 解析
- `<script async>`：下载不阻塞，下载完立即执行，顺序不保证
- `<script defer>`：下载不阻塞，HTML 解析完后按顺序执行（最常用）

#### CI 流水线常见步骤

```
代码提交 → ESLint/Prettier 检查 → TypeScript 类型检查 → 单元测试 → 构建 → 部署
```

### 题目：浏览器缓存策略（Header 混淆）

#### 强缓存（不发请求）

| Header | 说明 |
|---|---|
| `Expires` | HTTP/1.0，绝对时间，已被 Cache-Control 取代 |
| `Cache-Control: max-age=31536000` | 相对时间（秒） |
| `Cache-Control: no-cache` | 不用强缓存，走协商缓存 |
| `Cache-Control: no-store` | 完全不缓存 |
| `Cache-Control: public/private` | 允许/不允许中间代理缓存 |
| `Cache-Control: immutable` | 资源永不过期（适合带 hash 的文件） |

#### 协商缓存（发请求，可能返回 304）

方案一（修改时间）：
- 响应头：`Last-Modified`
- 请求头：`If-Modified-Since`

方案二（内容指纹，更精确）：
- 响应头：`ETag`
- 请求头：`If-None-Match`

服务器返回 304（无 body，用本地缓存）或 200（新资源）

#### 完整判断流程
```
浏览器请求资源
  → 有强缓存且未过期？ → 是 → 200 from cache（不发请求）
  → 否 → 发请求，走协商缓存
  → 发送 If-Modified-Since / If-None-Match
  → 服务器返回 304 或 200
```

### 题目：CORS 跨域原理（不了解）

#### 关键响应头
```
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: Content-Type, Token
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 86400
```

#### 简单请求（直接发）
条件：GET/POST/HEAD + 标准 Content-Type + 无自定义 Header
```
浏览器直接发请求（带 Origin 头）→ 服务器返回 Allow-Origin → 浏览器检查
```

#### 预检请求（先发 OPTIONS）
不满足简单请求时（如 JSON Body、自定义 Header、PUT/DELETE）：
```
浏览器先发 OPTIONS（带 Request-Method、Request-Headers）
    → 服务器返回允许的配置
    → 浏览器检查通过 → 再发真正的请求
    → 浏览器检查不通过 → 直接报错
```

#### 其他跨域方案
- **JSONP**：`<script>` 标签不受限，只支持 GET
- **Nginx 反代**：服务端转发，浏览器认为同源
- **postMessage**：iframe 窗口间通信
- **WebSocket**：协议不受同源限制

### 题目：Web 安全 XSS/CSRF/CSP（理解不精确）

#### XSS（跨站脚本攻击）
核心：向页面注入恶意脚本，在用户浏览器中执行。

三种类型：
- **存储型**：恶意脚本存入数据库，所有用户执行（论坛帖子）
- **反射型**：恶意脚本在 URL 参数中，服务器拼接到页面返回
- **DOM 型**：前端 JS 直接将用户输入拼接到 DOM（innerHTML）

防御：输入转义、HttpOnly Cookie、用 textContent 替代 innerHTML、CSP

#### CSRF（跨站请求伪造）
核心：不是获取 cookie，而是利用浏览器自动携带 cookie 的特性。
```
用户登录 bank.com → 访问 evil.com → evil.com 发起 bank.com 请求
→ 浏览器自动带 cookie → bank.com 以为是本人
```

防御：CSRF Token、SameSite Cookie（Strict/Lax）、验证 Referer/Origin、JWT

#### CSP（内容安全策略）
通过 HTTP 头限制资源加载：
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-abc';
```

## 模块 6：算法与手写题

### 题目：防抖和节流实现（有 Bug）

#### 两个共同问题
1. `fn.arguments` 是废弃属性，此时为 undefined，应直接用 `args`
2. `fn.apply(null, ...)` 导致 this 丢失，应保存 this 引用

#### debounce 正确实现
```typescript
function debounce(fn: Function, delay: number) {
  let timer: ReturnType<typeof setTimeout> | null = null
  return function(this: unknown, ...args: unknown[]) {
    if (timer) clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}
```

#### throttle 正确实现（带尾部执行）
```typescript
function throttle(fn: Function, duration: number) {
  let timer: ReturnType<typeof setTimeout> | null = null
  let lastTime = 0
  return function(this: unknown, ...args: unknown[]) {
    const now = Date.now()
    const remaining = duration - (now - lastTime)
    if (remaining <= 0) {
      if (timer) { clearTimeout(timer); timer = null }
      lastTime = now
      fn.apply(this, args)
    } else if (!timer) {
      timer = setTimeout(() => {
        lastTime = Date.now()
        timer = null
        fn.apply(this, args)
      }, remaining)
    }
  }
}
```

#### 关键区别
- 防抖：连续触发只执行最后一次
- 节流：固定间隔执行，带尾部执行确保最后一次不丢失

### 题目：手写 call/apply/bind（多个 Bug）

#### 常见问题
1. 参数未展开：`fn(params)` → 应为 `fn(...params)`
2. 缺少返回值
3. 属性名冲突：应用 `Symbol()` 作为临时 key
4. `typeof null === 'object'`，null 检查应用 `== null`
5. 原始值应用 `Object()` 包装
6. 应挂在 `Function.prototype` 上

#### call 实现
```typescript
Function.prototype.myCall = function(thisArg, ...args) {
  if (thisArg == null) thisArg = globalThis
  thisArg = Object(thisArg)
  const key = Symbol()
  thisArg[key] = this
  const result = thisArg[key](...args)
  delete thisArg[key]
  return result
}
```

#### apply 实现
```typescript
Function.prototype.myApply = function(thisArg, args) {
  if (thisArg == null) thisArg = globalThis
  thisArg = Object(thisArg)
  const key = Symbol()
  thisArg[key] = this
  const result = args ? thisArg[key](...args) : thisArg[key]()
  delete thisArg[key]
  return result
}
```

#### bind 实现
```typescript
Function.prototype.myBind = function(thisArg, ...bindArgs) {
  const fn = this
  return function(...callArgs) {
    return fn.apply(thisArg, [...bindArgs, ...callArgs])
  }
}
```
